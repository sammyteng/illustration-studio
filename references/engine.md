# 内容配图引擎（共用方法论）

> 这是 33 套配图风格的共用大脑。任何一套风格在执行时都先读这份引擎，
> 再读自己 `references/styles/<slug>.md` 里的「长相」。
>
> **一句话哲学**：好配图不是给文章配气氛图，也不是把文章画成流程图，而是
> **把文章里最关键的那个「认知动作 / 用户处境」挑出来，用一个画面让人 1 秒看懂**。

---

## 0. 这套引擎解决什么

把抽象内容（观点 / 流程 / 结构 / 状态 / 隐喻 / 用户处境）变成一张**清爽、有记忆点、一眼读懂**的配图。
通用流程**不随风格变**；只有「视觉 DNA + 是否有角色」随风格变（写在各 skill 自己的 `references/` 里）。

```
消化内容 → 提炼认知锚点 → 出 shot list → 选构图 → 发明原创隐喻
        → 写单图生图提示词 → 渲染（有 key 才渲，否则给可粘贴提示词）→ 质检双闸 → 交付
```

> 🌬️ **留白铁律（全局 CRITICAL · 所有风格无一例外）**：每张配图都必须留出**大片呼吸空间**——
> **主体 ≤ 60% 画面、≥ 30% 干净留白区，宁空勿满**。这是「认知配图」的命：画面一满就变成信息图 / 广告堆料，读者 1 秒看不懂、也压不住标题。
> 每条示例 / 渲染 prompt 都要**显式写** `generous negative space, lots of breathing room, uncluttered, airy`。
> Gate 1 逐张查留白；不留白即驳回重做。**即使是产品 hero / 大字广告这类天生满的风格，也要靠留白透气。**

---

## 1. 认知锚点提炼（最重要的一步，决定配图好坏）

先读用户给的正文 / 链接 / Markdown / 截图。**不要平均配图**——一篇文章不是每段都要配图。
只挑「认知锚点」：读者读到这里会有一个判断翻转、状态切换、或"这不就是我吗"的瞬间。

**4 类锚点 +「处境」法**：

| 锚点类型 | 长什么样 | 例子 |
|---|---|---|
| **核心判断 / 断言** | 一句反直觉或定调的话 | "信任不是喊出来的，是一块证据一块证据铺过去" |
| **状态转换** | 从 A 变成 B | 混乱→有序、手动→自动、卡住→跑起来 |
| **流程 / 结构** | 输入→处理→输出，分流，闭环 | 一条原始素材加工成流量/信任/转化三种内容 |
| **物理隐喻** | 抽象概念天然能映射成一个物件动作 | 信息过载 = 纸条不断飞出、有人拿纸板挡 |

**「处境」提炼五问**（小红书 / 共鸣类内容尤其用这个）：
1. 这段内容里读者是谁？（打工人 / 程序员 / 产品 / 创作者 / 一人公司 / AI 时代职场人）
2. 他正在被什么困住、拉扯、筛掉、压住、催促？
3. 他最熟悉的那句话是什么？
4. 看到图后他应该说「这不就是我吗」还是「这个观点好清楚」？
5. 哪 2-4 个短词能一秒戳中？

**核心冲突压缩句**（提炼到这一句就成功一半）：

```
谁，被什么，以什么方式，推向什么结果？
```

> 完成标记：已经明确「核心锚点 / 核心冲突 / 一句话画面」，且确认了哪些段落**不需要**图。

---

## 2. 先出 shot list（生图前的方案）

如果用户说「分析怎么配图 / 思考哪里需要图 / 先给方案」，**只出 shot list，不生图**。
默认 **4-8 张**；短文 1-3 张；长文也别轻易超过 9 张。够用就好，别把正文做成画册。

每张 shot 写清：

```
放在哪段后：
图的主题：
核心意思（一句话画面）：
认知锚点类型：
构图模式：
画面物件 / 动作：
建议中文短标注（2-8 字，最多 5-8 处）：
为什么这张值得画：
```

如果用户明确说「生成 / 出图 / 做图 / 看效果」→ **不要停下来等确认**，直接进入生图，但内部仍要先完成 shot list 思考。

---

## 3. 构图模式库（选一种，别混太多）

| 模式 | 适合 | 画法骨架 |
|---|---|---|
| **Workflow 流程** | 输入→处理→输出、内容生产、自动化链路 | 左输入 / 中处理 / 右输出，主流向用「强调色」箭头 |
| **系统局部** | 信息源、过滤器、数据库、agent 局部 | 只画 3-5 个核心模块，聚焦其中一个关键动作 |
| **前后对比** | 混乱/有序、手动/自动、焦虑/稳定 | 左混乱右稳定，中间一个转化箭头 |
| **角色 / 主体状态** | 用户痛点、工具太多、卡住到跑起来 | 2-4 个小状态，每个一个短标注 |
| **概念隐喻** | 内容工厂、信息仓库、脑内黑盒、自动日报 | 一个大的怪物件/机器，少量输入，一个输出，要有记忆点 |
| **方法分层** | 方法论框架、能力栈、系统层级 | 一层层盒子（别画正式金字塔） |
| **地图路线** | 从想法到上线、用户路径、学习路线 | 一条弯曲路径 + 少量节点 |
| **小漫画分镜** | 失败到成功、真实过程、使用前后 | 2-4 格，每格只表达一个动作 |

> 规则：**结构要自然表达，不要在图上写「Workflow / 系统架构图 / 常见坑」这种类型标题**。

---

## 4. 原创隐喻三步法（防止千篇一律 / 防止照抄案例）

每次都从**当前这篇文章**重新发明隐喻，不能套用旧图。

1. **抽象概念 → 物理动作**：卡住、漏掉、变重、分拣、沉淀、发酵、打结、拽回、压住、筛掉、盖章、缝上、称重。
2. **系统结构 → 低科技物件**：坏机器、纸箱、抽屉、漏斗、秤、邮筒、门、井、梯子、水管、线团、闸门、转盘、黑盒。每次只选 1-2 个，别堆满。
3. **让主体承担动作**（无固定主角时，由「真实物件 + 通用小场景」承担）：不是把物件摆好看，而是让画面里发生一个**物理冲突**——被拉、被挡、被推、被筛、被压、被重命名。

**反照抄铁律**：参考案例只用来校准「线条密度 / 留白 / 颜色克制」，**绝不复用**案例的物件组合、空间关系、标签位置。
同类主题也要换新隐喻（"承接路径"不一定画路线，可以画把内容尾巴接到门把手上）。

---

## 5. 事实锚定（涉及真人 / 品牌 / 数字时强制）

涉及个人经历、品牌名、公司名、粉丝数、时间跨度、成绩数字时：
- 只能用**用户输入 / 用户提供素材 / 用户明确确认**的事实。
- 用户没给的，**不要从案例继承、不要补成更完整的履历**。
- 无法确认的，用概括标签替代（"内容平台""项目节点""用户反馈"）。
- 用户要求保留但信息不全 → 在交付说明里标「待确认」，不要画成确定事实。
- **绝不加入用户未授权的真人姓名、联系方式、Logo、隐私信息。**

---

## 6. 生图 → 渲染路径

把 shot 拼成一条**单图生图提示词**（一次一张，不要把多张拼一张）。
配好至少一个出图 key 即可直接渲染 PNG；无 key 时只产出「可粘贴提示词」。

**最省事：调用本仓库自包含的渲染器** `render.js`（先 `npm install openai @google/genai`）：

```bash
node render.js --backend both --aspect 4:3 --out /tmp/illus \
  --prompt "<英文画面主体 + 中文标注词>"
# --backend gpt|nano|both ; --aspect 4:3|3:4|1:1 ; --prompt-file <path> 亦可
```

两个后端的凭证（render.js 已内置，手动调时照此）：

| 后端 | 模型 | 凭证 / 关键点 |
|---|---|---|
| **A · gpt-image-2** | OpenAI 兼容 | `OPENAI_API_KEY`（用官方 OpenAI；自有兼容路由再设 `OPENAI_BASE_URL`）。`render.js --backend gpt`。size：16:9→1536x1024 / 3:4→1024x1536 |
| **B · Gemini** | Nano Banana Pro | `GEMINI_API_KEY`（或 `GOOGLE_API_KEY`）。`render.js --backend nano`，model `pro`=gemini-3-pro-image-preview，失败降级 `flash` |

**默认后端选择规则**：
- **图上有文字标注 → 用 A · gpt-image-2**（中文/英文文字渲染最准，几乎不乱码）。
- **纯画面、无文字 → 用 B · Gemini Nano Banana Pro**（线条/质感更精细、成品感更强）。
- `render.js` 已支持 `--has-text true|false` 按此自动选后端；显式 `--backend` 优先。

观感差异（择一时参考）：gpt-image-2 更克制利落、更"听话"；Gemini 更精细、成品感更强。
出图后做图像级质检（第 7 节）。

- **缺 key 兜底**：万一两后端 key 都不可用，**不报错**，交付「shot list + 可直接粘贴提示词」，状态 `DONE_WITHOUT_RENDER`，提示用户粘到 ChatGPT / Gemini / 即梦 出图。

> **Token / key 来源**：永远从环境变量或 Keychain 读，**绝不把任何 key 写进 SKILL.md / 提示词 / render.js**。

## 7. 质检双闸门（不过不交付）

生成提示词后**先自检提示词**；渲染出图后**再自检图像**。各 skill 的 `references/qa-checklist.md` 补充该风格的专属红线。

### Gate 1 · CRITICAL（不过 = 不能当最终交付）
- [ ] 锁定了**单一**认知锚点，一张图只讲一个核心结构（没有"元素清单化"——把主题里所有名词都堆进去）。
- [ ] 用了**原创隐喻**，不是照抄参考案例（换主物件 / 换动作 / 换空间关系）。
- [ ] 提示词包含该风格 DNA 的**全部必含项**（背景 / 线条或材质 / 配色语义 / 比例 / 留白）。
- [ ] 中文短标注 **≤ 该风格上限**，每处短、能读、来自真实语境（不是"提升协作效率"这种概念词）。
- [ ] 比例默认 **4:3**（除非用户特别指定其它比例）。
- [ ] 事实锚定通过：没有用户未授权的真人/品牌/数字。
- [ ] 画面**3 秒读懂**：如果必须靠文字解释隐喻才懂，不合格。

### Gate 2 · STANDARD（不过 = 标注后可继续迭代）
- [ ] 留白充足、主体不超过画面约 60%。
- [ ] 配色克制（强调色只做语义，不抢主体）。
- [ ] 渲染出图时：物件 / 线条质感统一，无截图感、无贴图边界、无 PPT 味。
- [ ] 交付说明能指出「哪张最稳、哪张可选 / 需再收一轮」。

### 自循环（Loop 化）
执行 → 按 Gate 1 自检 → 不过则定位问题（太满 / 太像 PPT / 锚点不清 / 标注超量 / 照抄案例）→ 重写提示词 → 再自检。
**≥3 轮仍不过 Gate 1** → 走第 8 节失败回执，并把当前最佳版本标 `[TODO]` 返回，写清卡在哪。

### 交付状态
| 状态 | 含义 |
|---|---|
| `DONE` | shot list / 提示词 / （有 key 时）PNG 全部通过质检 |
| `DONE_WITHOUT_RENDER` | 提示词通过质检，但本机无 key 未渲染（已给可粘贴提示词）|
| `DONE_WITH_CONCERNS` | 已交付，但有文字稳定性 / 物件真实感 / 留白等待优化项 |
| `BLOCKED` | 缺主题/正文，或用户要求的事实无法确认 |

## 8. 失败回执（不许静默退出）

≥3 轮自检仍不过，或缺关键输入：

1. **写日志**：`~/.claude/logs/illustration-<skill>-<YYYYMMDD-HHMM>.md`，记录：输入摘要 / 卡在哪条 Gate / 已试方案 / 建议下一步。
2. **推送飞书**：尝试用 `lark-im` skill 给「小诗」发一条 DM（"配图 skill <name> 卡住，详见日志 <path>"）。推送失败不阻塞，但要在返回里说明。
3. **返回用户**：明确说「没过 = 没过」，给出当前最佳 shot list / 提示词 + 卡点，不要包装成完成。

---

## 9. 输出口径

- 生成前：短而准，给 shot list 或生成计划。
- 生成后：说清生成/规划了几张、每张用途、（有图时）保存路径、哪些最稳哪些可选。
- **别长篇讲风格理论，让图自己说话。**
